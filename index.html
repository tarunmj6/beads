01 — Setup

Setting Up Beads in a Project

Each project gets its own Beads database. The setup is a one-time process per project.

Project Initialization Flow
Startnew project
→
cd your-projectnavigate to folder
→
bd initcreates .beads/
↓
bd setup claudeor cursor / aider
→
git add .beads/
git commitversion control it
→
Ready!agents can now use bd
bash — one-time setup
# Step 1: Go to your project
cd ~/projects/my-app

# Step 2: Initialize Beads (creates .beads/ folder)
bd init

# Step 3: Connect to your AI agent (pick one)
bd setup claude    # Claude Code → writes CLAUDE.md
bd setup cursor    # Cursor IDE  → writes .cursor/rules/beads.mdc
bd setup aider     # Aider       → writes .aider.conf.yml

# Step 4: Commit the database to git
git add .beads/
git commit -m "Initialize beads issue tracker"

# That's it! Verify it works:
bd status
💡

Key idea: The .beads/ folder lives inside your project and gets committed to git. This means every agent session — even on a different machine — can pick up right where it left off.

02 — The Core Loop

How the Agent Loop Works

Once Beads is set up, your AI agent follows a tight, repeatable loop. It never needs to guess what to do next.

🔍 Check for ready workbd ready --json
↓
🎯 Claim the top-priority taskbd update <id> --claim --assignee agent-1
↓
⚙️ Do the workwrite code, tests, docs…
↓
✅ Mark it donebd update <id> --status closed
↓
🔄 Loop back — check for morebd ready --json
↺ repeat until bd ready returns empty
bash — agent loop commands
# Get all tasks with no blockers (ready to work)
bd ready --json

# Claim a task (sets status → in_progress + assigns you)
bd update feat-a1b2 --claim --assignee agent-1

# See full task details
bd show feat-a1b2

# Add a comment/note as you work
bd comment feat-a1b2 "Implemented the login form, writing tests now"

# Mark done
bd update feat-a1b2 --status closed

# Check again for next task
bd ready --json
03 — Solo Workflow

You + One Agent

The most common use case: you describe a feature, your agent breaks it into tasks and executes them.

Solo Example: Building a User Auth System

  1. You create an Epic (high-level goal)

    An epic is the big goal. Tasks are the steps to get there.

    bash
    bd create "User Authentication System" \
  -t epic \
  -p 1 \
  --description "Full auth with login, signup, JWT tokens, password reset"

  2. Agent breaks it into tasks with dependencies

    You tell your agent: "Read the epic and create subtasks for everything needed." The agent then files them with proper dependencies.

    bash — what the agent runs
    # Agent creates tasks...
bd create "Set up database schema for users" -t task -p 1  # → auth-001
bd create "Build signup endpoint" -t task -p 1             # → auth-002
bd create "Build login endpoint + JWT" -t task -p 1          # → auth-003
bd create "Password reset flow" -t task -p 2               # → auth-004
bd create "Write integration tests" -t task -p 2            # → auth-005

# Agent sets dependencies (login needs DB first)
bd dep add auth-002 --blocks auth-001  # signup blocked by DB
bd dep add auth-003 --blocks auth-001  # login blocked by DB
bd dep add auth-005 --blocks auth-003  # tests blocked by login

  3. Agent executes in dependency order

    bd ready automatically returns only tasks whose blockers are done — so the agent always does things in the right order.

    bash
    bd ready
# Returns: auth-001 "Set up database schema" ← only unblocked task

# Agent does the work, then closes it
bd update auth-001 --status closed

bd ready
# Now returns: auth-002, auth-003 ← both unblocked now

  4. You check in whenever you want

    bash
    bd list              # See all tasks + status
bd status            # Summary: open/in-progress/done counts
bd dep tree auth-001 # Visualize dependency graph
04 — Multi-Agent Workflow

Multiple Agents, One Shared Database

This is where Beads really shines. Multiple agents work in parallel, each claiming different tasks, with zero conflicts.

🤖 Agent 1 — Frontend

Build login form
Add form validation
Write UI tests
📿 .beads/
Shared DBvia git / BEADS_DIR
↔

🤖 Agent 2 — Backend

Set up DB schema
Build auth endpoints
Add rate limiting
bash — multi-agent setup
# Both agents use the same project's .beads/ via git
# Agent 1 terminal:
cd ~/my-app
bd ready --json
bd update feat-001 --claim --assignee agent-frontend

# Agent 2 terminal (same project, different terminal):
cd ~/my-app
bd ready --json
# feat-001 is already claimed → won't be returned
# Agent 2 automatically gets feat-002 instead
bd update feat-002 --claim --assignee agent-backend

# See what each agent is working on
bd list --assignee agent-frontend
bd list --assignee agent-backend
⚡

How conflicts are avoided: The --claim flag is atomic — it sets status: in_progress and assignee in a single operation. So two agents can never claim the same task simultaneously.

Full Example Building a Redis Caching Layer

  1. Phase 1: Brainstorm with Superpowers

    You kick off a conversation. Superpowers intercepts and asks clarifying questions before any code is written.

    Claude Code session
    You:      I want to add caching to our API

Superpowers: Let me ask a few questions first...
              - What are you caching? (sessions, query results, user data?)
              - Scale needs? (single instance or distributed?)
              - TTL requirements?
              - Redis or in-memory?

You:      Session data, distributed, 24hr TTL, Redis

Superpowers: Got it. I recommend ioredis over node-redis for
              cluster support. Shall I write a plan?

  2. Phase 2: Write & Review the Plan

    Superpowers writes a detailed plan — you read it, edit it, add constraints. This is your chance to steer before any code exists.

    Claude Code session
    Superpowers: Here's the plan (saved to PLAN_redis_cache.md):

  Task 1: Add ioredis + types
  Task 2: Create src/cache/redis.ts service
  Task 3: Replace SessionStore with RedisSessionStore
  Task 4: Integration tests (real Redis, not mocked)
  Task 5: Update Docker Compose to include Redis

You:      Good, but add: key format must be session:{userId}:{id}
              and never cache auth tokens directly. Update Task 2.

Superpowers: Updated. Ready to convert this to Beads epics?

  3. Phase 3: Convert Plan → Beads Tasks

    Agent translates the approved plan into Beads with full context embedded in each task description.

    bash — what the agent runs
    bd create "Redis Caching Layer" -t epic -p 1 \
  --description "Distributed session cache via Redis/ioredis.
Key format: session:{userId}:{sessionId}. TTL: 24h.
Do NOT cache auth tokens."
# → cache-001 (epic)

bd create "Add ioredis dependency" -t task -p 1 --parent cache-001
bd create "Create redis.ts cache service" -t task -p 1 --parent cache-001
bd create "Replace SessionStore with RedisSessionStore" -t task -p 1 --parent cache-001
bd create "Integration tests (real Redis)" -t task -p 2 --parent cache-001
bd create "Add Redis to Docker Compose" -t task -p 2 --parent cache-001

# Set dependency order
bd dep add cache-003 --blocks cache-002  # redis.ts must exist first
bd dep add cache-004 --blocks cache-003  # tests need store to exist

  4. Phase 4: Execute with Subagents

    Superpowers spawns isolated subagents per task. Each starts with clean context. A code reviewer validates before closing.

    Claude Code session
    You:      /superpowers:execute-plan

Superpowers: Starting subagent for cache-002 (Add ioredis)...
              ✓ Done. Code reviewer passed.
              bd update cache-002 --status closed

              Starting subagent for cache-003 (redis.ts)...
              ✓ Done. Code reviewer passed.
              bd update cache-003 --status closed

              Starting subagent for cache-004...
              ⚠ Code reviewer flagged: auth token being cached!
              Sending back to executor for fix...
              ✓ Fixed. bd update cache-004 --status closed
🧠

The key distinction: Superpowers is where you talk to the agent about what and how. Beads is what survives that conversation — the structured, queryable record that future sessions and other agents can act on without re-explaining anything.

Interrupt Example Agent taking the wrong approach — stop & redirect
bash — interrupt sequence
# Agent is in_progress on cache-003 but using the wrong library

# Step 1: Stop it — set back to open (unclaims it)
bd update cache-003 --status open

# Step 2: Leave a comment explaining the redirect
bd comment cache-003 "STOP — do not use node-redis.
Use ioredis instead (supports cluster mode).
See cache-002 for the installed package.
Key format MUST be session:{userId}:{sessionId}"

# Step 3 (optional): Rewrite the description with the corrected approach
bd update cache-003 \
  --description "Create src/cache/redis.ts using ioredis (NOT node-redis).
Implement RedisSessionStore class.
Key format: session:{userId}:{sessionId}
TTL: 86400 seconds (24h)
Do NOT cache auth tokens under any circumstance."

# Agent picks it up next session, reads comments + new description
bd show cache-003  # verify it looks right before resuming
Discussion Example Embedding a full design discussion into a task

Use this pattern when you've had a long discussion and want future agents (or yourself) to have the full context without re-explaining.

bash — embed discussion as task description
bd update cache-003 --description "$(cat <<'EOF'
GOAL: Redis session cache service

DISCUSSION NOTES (agreed 2026-03-08 with human):
- Use ioredis (not node-redis) — cluster support required
- TTL: 86400s (24h) — non-negotiable per security policy
- Key format: session:{userId}:{sessionId}
- NEVER cache auth tokens — store them in httpOnly cookies only

APPROACH:
1. src/cache/redis.ts — RedisSessionStore class
2. connect via REDIS_URL env var (fallback: localhost:6379)
3. Methods: get(key), set(key, val, ttl), del(key)
4. Wrap all ops in try/catch, log errors but don't throw

OPEN QUESTIONS (decide before implementing):
- Redis Cluster vs Sentinel? (ask human if unsure)
- Should we use a connection pool? (default ioredis handles this)

VERIFICATION:
- Run: docker compose up redis
- Run: npm test -- --testPathPattern=redis
- Confirm TTL is set: redis-cli TTL session:test:abc
EOF
)"
💬

Use bd comment for running notes, bd update --description for the source of truth. Comments are a thread (permanent log). Description is what the agent reads as its brief. Keep the description clean and current — use comments for history.

Handoff Example Handing a task from one agent to another
bash — agent handoff
# Agent 1 got stuck / context too long — hand off to Agent 2

# Leave a detailed handoff comment

- Next step: replace MemoryStore with RedisSessionStore, run tests

# Reassign to new agent

# Agent 2 reads the full context

09 — Tips

Best Practices

🏷️

Use short prefixes

When initializing, use a short prefix like auth- or ui- to keep IDs readable: bd init --prefix auth

📏

Keep issues small

Aim for under 200 open issues per project. Run bd admin compact --days 30 every few days to summarize old closed tasks.

🔁

Run bd doctor daily

Especially after git merges. Run bd doctor --fix to catch and fix any database inconsistencies.

📝

Always commit .beads/

After every session, run git add .beads/ && git commit -m "Update beads" so nothing is lost between sessions.

🎯

Epics first, tasks second

Create the epic with a detailed description, then let your agent decompose it into tasks. Better planning = fewer orphaned tasks.

👁️

bd list is your dashboard

Run bd list at the start of every session to re-orient yourself before giving any instructions to your agent.
